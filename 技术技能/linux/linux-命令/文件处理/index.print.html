  • 产品定位：以优质非转基因甜玉米为基底，开发健康导向的休闲糖果，主打“真玉米味、少添加、有嚼劲/有酥感”，区别于传统明胶软糖或高糖硬糖。
  • 核心形态（初期双线开发）：
    ✅ 玉米酥粒糖（烘烤型）：酥脆微甜，含玉米胚芽粉+海藻糖+天然玉米汁浓缩液，0反式脂肪，可作零食/佐餐小食；
    ✅ 玉米清甜软糖（胶质型）：以玉米淀粉糖浆+果胶+天然玉米黄质着色，不添加人工色素/香精，pH值稳定防粘连。

  • Linux命令

创建测试文件

fallocate

1# 命令语法
2fallocate -l 1G test_fallocate.log

参数说明：

  • -l 1G ：指定文件长度为1GB（也可用 1M、1K或字节数如 1073741824）
  • test_fallocate.log ：生成的文件名

特点:

  • 速度极快（<0.1秒），直接分配磁盘空间，实际占用大小 = 文件大小
  • 支持的文件系统：ext4、xfs、btrfs 等主流本地文件系统

验证：

1ls -lh test_fallocate.log
2du -h test_fallocate.log

注意：

  • 在tmpfs、NFS、FUSE、ext3等文件系统上可能报错 operation not supported
  • 生成的文件内容为空洞（未初始化，读出来可能为零，但不会触发磁盘I/O分配）

dd

真实写入数据，速度较慢

1# 命令语法
2dd if=/dev/zero of=test_dd.log bs=1M count=1024 status=progress
3dd if=/dev/urandom of=test_dd.log bs=1M count=1024 status=progress

参数说明：

  • if=/dev/zero 输入文件为/dev/zero（无限输出零字节）
  • of=test_dd.log 输出文件
  • bs=1M 每次读写1MB的块大小（调大可略微提速）
  • count=1024 总共写入1024个块 -> 1024MB
  • status=progress 实时显示已传输的字节数

特点：

  • 真正执行写入操作，耗时与磁盘速度相关（机械硬盘约10-20秒，SSD几秒）
  • 生成的文件实际占用空间 = 1GB，且内容全为0

验证：

1ls -lh test_dd.log
2du -h test_dd.log

注意：

  • 若不指定count，dd会一直读到输入文件结束，需手动中断
  • 使用 /dev/urandom可生成随机内容：if=/dev/urandom，但速度更慢

truncate

创建稀疏文件（瞬间完成，不占实际空间）

1# 命令语法
2truncate -s 1G test_truncate.log

参数说明：

  • -s 1G 设置文件大小为1GB（支持K、M、G、T）
  • test_turncate.log 文件名

特点：

  • 速度极快，几乎不消耗磁盘空间（仅占用少量元数据）
  • 文件看起来是1GB，但实际占用几乎为0

验证：

1ls -lh test_truncate.log #显示1G
2du -h test_truncate.log  # 显示0

注意：

  • 若向该文件写入数据（例如用dd覆盖某段区域），会逐渐占用真实空间
  • 某些就程序（如scp、rsync）传输稀疏文件时可能自动扩展为完整大小，导致传输量剧增
  • 所有主流文件系统（ext4、xfs、btrfs、tmpfs）都支持稀疏文件

如何选择

场景 推荐命令
快速生成一个大文件占位，用于测试路径、脚本逻辑 fallocate或truncate
需要真实占用磁盘空间（如I/O性能、磁盘容量） fallocate
需要确保文件内容有实际数据 dd
文件系统不支持fallocate dd 或 truncate

如果需要进一步验证文件内容是否全为零，可以用下面命令查看

1hexdump -C test_fallocate.log | head

软链接和硬链接

软链接

软链接是一个特殊的独立文件，它的内容是目标文件的路径字符串

  • 软链接有自己的inode和文件类型l。
  • 访问软链接时，系统会读取其内容，然后跳转到目标路径。

创建与删除

1ln -s 目标路径 软连接名
  • 删除软链接本身：rm softlink，不影响目标
  • 删除目标文件，软链接变成“悬空链接”（指向不存在的路径），访问时报错。

特点与限制

  • 可以跨文件系统（因为只存储路径字符串）
  • 可以对目录创建软链接
  • 可以指向不存在的路径（不检查目标是否存在）
  • 软链接的权限（lrwxrwxrwx）是固定的，实际访问时取决于目标文件的权限
  • 占用少量空间（存储路径字符串）

硬链接

硬链接是多个文件名指向同一个inode

  • 创建硬链接后，新文件名与旧文件名共享相同的inode编号
  • 硬链接计数会增加

创建与删除

1ln 源文件 硬链接名

删除任意一个文件名，只要还有至少一个文件名指向该inode，文件数据就不会被真正删除。只有当硬链接计数为0时，磁盘空间才被释放。

特点与限制

  • **不能跨文件系统：**因为inode编号只在各自文件系统内唯一
  • **不能对目录创建硬链接：**防止循环引用，虽然系统底层允许但普通用户禁止
  • 修改任意一个链接，所有链接看到的内容都相同（因为共享数据块）
  • 不占用额外空间（只增加一条目录项，约几个字节）
  • 硬链接文件与源文件的文件类型、权限、所有者、时间戳完全一致

对比表格

特性 硬链接 软链接
inode 与源文件相同 自己的独立inode
文件类型 普通文件 符号链接
跨文件系统 不可以 可以
指向目录 不可以 可以
目标删除后 数据仍存在 链接失效
大小 仅目录项 存储路径字符串的大小
权限 与源文件完全相同 固定0777，实际由目标决定
相对路径 不支持 支持
查找所有链接 通过find -samefile 或 inode号 无法直接反向查找

使用场景

硬链接适用场景

  • 备份/快照 节省空间
  • 程序文件版本管理（多个名字指向同一可执行文件）
  • 需要多个路径访问同一份数据，且不关心文件是否被“移动”

软链接适用场景

  • 快捷方式（跨分区、跨文件系统）
  • 动态库版本管理
  • 链接到目录
  • 需要指向可能变化的路径

注意

  • 硬链接 != 复制：复制是新的inode + 新的数据块，硬链接共享数据块

  • 软链接相对路径问题：相对路径是相对于软链接所在目录解析，而不是当前工作目录

  • 删除原始文件

    • 硬链接：数据还在，其他链接可用
    • 软链接：变成断链

  • 目录硬链接：除非使用mount --bind，否则不要尝试。系统不允许，因为会导致目录循环和rm -fr 无限递归。
