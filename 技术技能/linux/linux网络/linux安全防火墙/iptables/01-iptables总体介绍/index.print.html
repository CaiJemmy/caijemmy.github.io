iptables总体介绍

iptables 是 Linux 内核中 netfilter 框架的用户态管理工具，其“执行过程”本质上是指数据包在内核网络协议栈中经过 netfilter 钩子（hook）时，如何被 iptables 规则匹配与处理。需注意：iptables 命令本身不“执行”数据包处理，而是配置规则；真正的匹配与动作发生在内核 netfilter 的钩子点上。以下是清晰、准确的执行过程说明（基于 IPv4，标准内核流程）：

iptables 工具体系概述

iptables 是用户空间工具，通过 netlink 与内核 Netfilter 交互，管理 IPv4 规则；对应 IPv6 的工具为 ip6tables（已逐步被 nftables 统一替代）。

1、表（Tables）与链（Chains）

iptables 将规则组织在不同**表（table）**中，每张表专注一类功能，各表包含预定义链（chains）：

表名 主要用途 默认链（IPv4）
filter 包过滤（防火墙核心） INPUT、FORWARD、OUTPUT
nat 网络地址转换（DNAT/SNAT/MASQUERADE） PREROUTING、OUTPUT、POSTROUTING
mangle 修改数据包头（TTL、TOS、MARK等） PREROUTING、INPUT、FORWARD、OUTPUT、POSTROUTING
raw 绕过连接跟踪（NOTRACK） PREROUTING、OUTPUT
security 强制访问控制（SELinux 相关，较少使用） INPUT、FORWARD、OUTPUT

注：各表的链在 Netfilter hook 上按固定优先级顺序注册（如 raw → mangle → nat → filter），但同一 hook 上不同表的链是串行执行的，且规则按顺序匹配。

2、规则结构（Rule）

一条典型规则示例：

1iptables -t filter -A INPUT -s 192.168.1.100 -p tcp --dport 22 -m state --state NEW -j ACCEPT
  • -t filter：指定表；
  • -A INPUT：追加到 INPUT 链；
  • -s 192.168.1.100：源地址匹配；
  • -p tcp --dport 22：协议+目标端口匹配；
  • -m state --state NEW：使用 state 模块匹配连接状态；
  • -j ACCEPT：目标动作（ACCEPT、DROP、REJECT、LOG、自定义链等）。

3、匹配与动作机制

  • 匹配（Matches）：条件判断（如地址、端口、协议、状态、字符串、速率等），可组合（-m multiport --dports 22,80,443）；
  • 目标（Targets）：
    • 终止型：ACCEPT、DROP、REJECT（返回 RST/ICMP）；
    • 跳转型：JUMP 到用户自定义链（支持规则分组与复用）；
    • 扩展目标：LOG（记录日志）、NFLOG（发往 userspace）、REDIRECT（端口重定向）、MASQUERADE（动态 SNAT）等。

iptables 规则匹配与动作执行流程（单条链内）

当数据包到达某 hook 点（如 INPUT），内核按以下逻辑处理该链的规则：

  1. 顺序匹配：从链中第 1 条规则开始，逐条比对；
  2. 条件匹配：检查包的源/目的 IP、端口、协议、状态（conntrack）、接口、TCP 标志等是否满足规则条件；
  3. 匹配成功：
    • 若目标（target）为 ACCEPT / DROP / REJECT / LOG 等内置动作 → 立即终止该链匹配，执行对应动作（如丢弃或放行）；
    • 若目标为 自定义链 → 跳入该链继续匹配；返回后继续执行下一条规则（除非自定义链中明确 RETURN 或终止动作）；
    • 若目标为 SNAT / DNAT / MASQUERADE（仅在 nat 表特定链）→ 执行地址转换，并继续后续 hook 链（nat 表不终止流程，仅修改包）；
  4. 匹配失败：尝试下一条规则；
  5. 所有规则都不匹配 → 执行该链的 默认策略（policy）（如 iptables -P INPUT DROP）。

关键原则：

  • ACCEPT/DROP/REJECT 是终止动作（不继续后续规则）；
  • SNAT/DNAT 是修改动作，不终止匹配（但仅在 nat 表的 PREROUTING/OUTPUT/POSTROUTING 中生效）；
  • RETURN 用于从自定义链返回父链，继续执行父链的下一条规则。

典型场景示例（本机访问外网 Web 服务器）

 1本机发出请求（curl http://example.com）：
 21. 本地进程 → 协议栈 → NF_INET_LOCAL_OUT  
 3   ├─ raw OUTPUT → mangle OUTPUT → nat OUTPUT（可做源地址转换）→ filter OUTPUT  
 4   └─ 若 filter OUTPUT 允许 → 进入路由决策（查路由表）  
 52. 路由确定出口网卡（如 eth0）→ NF_INET_POST_ROUTING  
 6   └─ mangle POSTROUTING → nat POSTROUTING（如 MASQUERADE）→ 发送至网络  
 7
 8响应包返回本机：
 91. 入口网卡（eth0）→ NF_INET_PRE_ROUTING  
10   └─ raw PREROUTING → mangle PREROUTING → nat PREROUTING（DNAT）→ 连接跟踪确认为 ESTABLISHED  
112. 路由判断 → 目标是本机 → NF_INET_LOCAL_IN  
12   └─ mangle INPUT → filter INPUT（放行 ESTABLISHED）→ 交付给本地进程

运维与安全实践

  • 规则顺序至关重要：iptables 按链中规则从上到下顺序匹配，第一条匹配即执行对应动作（除非使用 -j CONTINUE 等特殊目标）。将高匹配频率规则（如 ESTABLISHED）放在链顶部；避免使用 -j LOG 在大量规则前（影响性能）；
  • 默认策略（Policy）：每条内建链有默认策略（ACCEPT 或 DROP），建议显式设置（如 iptables -P INPUT DROP），并确保管理通道（如 SSH）规则在前；
  • 状态化防火墙优势：结合 conntrack 和 -m state --state ESTABLISHED,RELATED -j ACCEPT，可安全放行返回流量，无需开放所有端口。或现代推荐的 -m conntrack --ctstate，依赖内核 nf_conntrack 模块，是高效防火墙的基础；
  • 持久化问题：iptables 规则重启后丢失，需通过 iptables-save / iptables-restore 或发行版工具（如 iptables-persistent、firewalld）保存；
  • 逐步迁移建议：Linux 内核 3.13+ 起，nftables 成为官方推荐替代方案（统一 IPv4/IPv6/ARP/bridge，语法更简洁，性能更优），但 iptables 仍广泛兼容（通过 iptables-nft 后端）。
  • 表与链的对应关系：
    • raw：绕过连接跟踪（NOTRACK）；
    • mangle：修改包头（TTL、TOS、MARK）；
    • nat：地址/端口转换（仅在 PREROUTING/OUTPUT/POSTROUTING）；
    • filter：核心过滤（INPUT/FORWARD/OUTPUT）；
    • security：SELinux/MAC 相关（较少使用）；
  • iptables 命令不实时处理包：iptables -A INPUT ... 仅向内核 netfilter 注册规则；包处理完全由内核在数据路径中完成，无用户态参与。

典型应用场景

  • 主机防火墙（限制 SSH、HTTP 访问源）；
  • 网关/NAT 设备（内网共享上网、端口映射）；
  • 流量限速与防攻击（limit、hashlimit 模块防暴力登录、CC）；
  • 日志审计（LOG 目标记录可疑连接）；
  • 透明代理/流量重定向（配合 REDIRECT 或 TPROXY）；
  • 容器/虚拟网络策略（如 Docker 默认使用 iptables 实现网络隔离）。
