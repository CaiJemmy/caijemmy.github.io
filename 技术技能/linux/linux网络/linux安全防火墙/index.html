Netfilter 是什么？（定义与定位）

Netfilter 是 Linux 内核中实现网络数据包过滤、地址转换（NAT）、连接跟踪（conntrack）、包修改等功能的 框架（Framework），不是用户直接操作的命令或工具。

  • 它是内核子系统（位于网络协议栈中），自 Linux 2.4 内核起成为标准组件，由 Rusty Russell 等人主导设计。
  • 它提供可扩展的钩子函数（hook points），允许内核模块（如 iptables 后端、nftables、xt_conntrack）在数据包穿越协议栈的关键路径上注册回调函数，实现策略干预。
  • 本质：Linux 网络栈的“策略注入点”，是所有 Linux 防火墙、NAT、负载均衡（如 IPVS）、容器网络（如 Docker、CNI）的底层基石。

类比理解：Netfilter ≈ “交通检查站的预留岗亭位置”，而 iptables/nftables ≈ “在岗亭里执勤的交警”（用户空间策略配置工具）。

TCP/IP参考模型中包过滤防火墙的位置

过滤入站数据包

1. 对远程源地址进行过滤

  • 通过数据包头中的源地址识别发送者，手段单一

  • 来自互联网包中源地址，以下为源地址欺骗和非法地址：

    场景1 场景2
    你自己的IP地址 回环接口地址 127.0.0.0~127.255.255.255
    你所在的局域网地址 畸形广播地址 0.0.0.0
    A、B、C类中的私有IP地址 A类网络的0地址 0.0.0.0~0.255.255.255
    D类多播IP地址 APIPA地址 169.254.0.0~169.254.255.255
    E类保留地址 网络测试地址 192.0.2.0~192.0.2.255

  • 阻止有问题主机、站点的访问

  • 允许特定远程主机和站点的访问

2. 对目标地址进行过滤

  • 主机防火墙：网络接口的地址；有限广播 255.255.255.255；子网广播 针对192.168.10.0/24 子网就是 192.168.10.255

  • 边界防火墙

    • 路由模式：内网主机的地址
    • NAT模式：外网接口的地址

注意：如果目标主机的IP是：0.0.0.0，很有可能是做linux主机的探测。BSD系统会对目标是0.0.0.0的数据包返回错误类型是3的ICMP消息。

3. 对远程源端口进行过滤

  • 远程客户端 => 本地服务器资源的访问：非特权范围内的端口（1024~65535）
  • 远程服务器 => 本地客户机的响应：服务器端口（如Web 80）

4. 对TCP连接状态进行过滤

  • 远程客户端 => 本地服务器资源的访问：第一个数据包，设置了SYN
  • 远程服务器 => 本地客户端的响应：每个包，设置了ACK

5. 刺探和扫描

  • 刺探（probe）：尝试连接到一个单独的服务端口，根据端口的响应作出判断
  • 扫描（scan）：对一台或多台主机的多个服务端口进行系列刺探
    • 通常自动执行
    • 扫描器之王 nmap

6. 拒绝服务攻击

  • 拒绝服务（Denial of Service，DoS）
  • 分布式拒绝服务（Distributed Denial of Service，DDoS）
  • 常见类型
    • ICMP Flood
    • SYN Flood
    • Teardrop Attacks
    • HTTP_Half_Connect
    • HTTP_Connect
    • UDP Flood
    • DNS Flood
    • … …

过滤出站数据包

  • 对本地源地址进行过滤
  • 对远程目标地址进行过滤
  • 对本地源端口进行过滤
  • 对远程目标端口进行过滤
  • 对出站TCP连接状态进行过滤

Netfilter 架构核心：5 个 Hook 点（钩子）

Netfilter 在 IPv4 协议栈中定义了 5 个预设的 hook 点（IPv6、ARP、DECnet 等也有对应版本），数据包按固定顺序经过这些点。每个 hook 是一个函数指针链表，可挂载多个规则模块。

Hook 名称 触发时机（数据包流向） 典型用途 是否可修改包内容
NF_INET_PRE_ROUTING 数据包刚进入网络栈，尚未进行路由决策（即不知是发给本机还是转发） • DNAT（目标地址转换）
• conntrack 初始化（首次见到新连接）		 ✅ 是
NF_INET_LOCAL_IN 数据包目的地是本机（路由后），即将交给上层协议（如 TCP/UDP） • 主机防火墙（INPUT 链）
• 限速、封禁 IP		 ✅ 是
NF_INET_FORWARD 数据包需转发给其他主机（非本机，且 ip_forward=1） • 转发防火墙（FORWARD 链）
• 网关/NAT 设备策略		 ✅ 是
NF_INET_LOCAL_OUT 本机主动发出的数据包，尚未路由（如 curl http://example.com） • 主机防火墙（OUTPUT 链）
• SNAT（源地址转换，较少在此做）		 ✅ 是
NF_INET_POST_ROUTING 数据包即将离开本机（路由已完成，确定出口设备），最后修改机会 • SNAT（最常用位置）
• 修改 TOS、TTL、出口 MAC（如 mangle）		 ✅ 是

关键规律：

  • 所有 “进入本机”的包：PRE_ROUTING → LOCAL_IN
  • 所有 “转发的包”：PRE_ROUTING → FORWARD → POST_ROUTING
  • 所有 “本机发出的包”：LOCAL_OUT → POST_ROUTING

Netfilter 的核心支撑模块

Netfilter 本身是框架，其能力依赖以下内核模块协同工作：

模块 功能说明 关联用户工具
nf_conntrack（连接跟踪） 为每个连接（TCP/UDP/ICMP 等）建立并维护状态条目（如 ESTABLISHED, RELATED, INVALID），是 state/conntrack 匹配的基础。所有 --state ESTABLISHED 规则都依赖它。 iptables -m state, nft ct state
nf_nat（网络地址转换） 提供 NAT 核心逻辑（SNAT/DNAT/MASQUERADE/REDIRECT），与 nf_conntrack 深度集成（NAT 后需更新连接跟踪状态）。 iptables -t nat, nft add rule ip nat
nf_tables（自 4.18+） 新一代通用规则表框架，替代旧 ip_tables，支持更高效匹配、原子提交、统一语法。nftables 的内核后端。 nft 命令
xt_* 系列模块（eXtended Tables） 扩展匹配（xt_multiport, xt_limit, xt_iprange）和目标（xt_LOG, xt_REJECT, xt_REDIRECT）的内核模块，按需加载。 iptables -m multiport, iptables -j LOG

注：iptables 是用户空间命令，其规则最终由 ip_tables 内核模块解析并注册到 Netfilter hook；而 nftables 使用更现代的 nf_tables 模块，性能与可维护性更优。

用户空间工具演进：iptables → nftables（Linux 4.18+ 推荐）

维度 iptables nftables
内核后端 ip_tables, iptable_nat, xt_* nf_tables（统一框架）
规则存储 每个表（filter/nat/mangle）独立内存结构 单一、可嵌套的规则集（tables/chains/rules）
性能 每条规则线性匹配，规则多时性能下降 使用高效查找结构（如 hash、radix tree），支持批量操作
语法统一性 iptables/ip6tables/arptables/ebtables 四套命令 nft 单一命令支持 IPv4/IPv6/ARP/bridge 等
原子性 iptables-restore 支持，但非默认 原生支持规则集原子替换（避免中间不一致状态）
现状 仍广泛使用，但已被标记为“legacy” Linux 官方推荐，新系统默认启用（如 RHEL 8+, Debian 11+, Ubuntu 20.04+）

提示：iptables 命令在新内核中通常通过 iptables-nft（兼容层）调用 nftables 后端，非原生 ip_tables。可通过 iptables -V 查看是否含 nf_tables。

典型应用场景（Netfilter 的实际价值）

场景 实现方式 依赖 Netfilter 特性
主机防火墙 nft add rule ip filter input ct state established,related accept nf_conntrack + NF_INET_LOCAL_IN hook
Docker 容器网络 docker0 网桥 + iptables -t nat -A POSTROUTING -s 172.17.0.0/16 ! -o docker0 -j MASQUERADE nf_nat + NF_INET_POST_ROUTING
Kubernetes Service（iptables 模式） 为每个 Service 生成 DNAT 规则，转发到 Pod IP nf_nat + NF_INET_PRE_ROUTING
端口映射（端口转发） nft add rule ip nat prerouting tcp dport 8080 dnat to 192.168.1.10:80 nf_nat + NF_INET_PRE_ROUTING
流量限速/限流 nft add rule ip filter output @netdevname meter httpout { ip saddr 10.0.0.0/8 limit rate 100 kbytes/second } xt_limit / nft meter + NF_INET_LOCAL_OUT
入侵检测日志 nft add rule ip filter input ip saddr 1.2.3.4 log prefix "BLOCKED:" drop xt_LOG / nft log + `NF_INET_LOCAL_IN
