01 哈希取余算法详解

一、算法原理

1. 基本概念

哈希取余算法是一种数据分片策略，通过对数据的哈希值进行取模运算来决定数据应该分配到哪个分区/节点。

2. 算法公式

1节点编号 = hash(key) % N

其中：

  • hash(key)：对键值进行哈希运算，得到一个整数
  • N：节点总数
  • %：取模运算符
  • 结果：0 到 N-1 之间的整数，表示数据应该存储的节点编号

3. 工作流程

 2
 3def hash_mod(key, node_count):
 4    hash_obj = hashlib.md5(str(key).encode())
 5    hash_int = int(hash_obj.hexdigest(), 16)
 6
 7    return hash_int % node_count
 8
 9nodes = 3
10data_items = ["user_001", "user_002", "user_003", "user_004", "user_005"]
11
12for item in data_items:
13    node = hash_mod(item, nodes)
14    print(f"数据 '{item}' -> 节点 {node}")

4. 哈希函数选择

常见的哈希函数：

  • MD5：128位，已不推荐用于安全，但分片可用
  • SHA-1：160位
  • SHA-256：256位，更均匀
  • MurmurHash：非加密哈希，性能好
  • CRC32：简单快速

02 一致性哈希算法详解

一致性哈希（Consistent Hashing）是一种在分布式系统中广泛应用的负载均衡与数据分片（sharding）技术，核心目标是：在节点（如缓存服务器、数据库分片、CDN节点等）动态增减时，尽可能减少数据重映射（rehash）带来的迁移开销。它由 David Karger 等人在 1997 年提出，已成为现代分布式架构（如 Redis Cluster、Cassandra、Dynamo、Kafka、负载均衡器等）的关键基础算法之一。

一、核心思想：从“离散映射”到“环形连续映射”

传统哈希（Modulo Hash） 一致性哈希（Consistent Hash）
node = hash(key) % N（N 为节点数） node = 顺时针最近的节点（在哈希环上）
节点增减 → 全量重哈希：N→N+1 时，约 (N+1)/N ≈ 100% 的 key 映射关系改变 节点增减 → 局部重映射：仅影响新节点与前一节点之间的 key，迁移比例 ≈ 1/N（理想情况下）

本质突破：将“节点数量”从哈希计算中解耦，使映射关系对节点规模变化具有鲁棒性。

二、算法原理详解（含虚拟节点优化）

步骤 1：构建哈希环（Hash Ring）

  • 定义哈希空间：通常为 [0, 2^32) 的整数环（类比钟表 0~360°），首尾相连。
  • 选择均匀性好、抗碰撞强的哈希函数（如 MD5、SHA-1、MurmurHash3），将任意输入映射到该空间。

步骤 2：映射物理节点（含虚拟节点）

  • 物理节点 → 哈希环：对每个物理节点 node_i，计算其哈希值 h(node_i)，标记在环上。
  • 引入虚拟节点（Virtual Nodes）（关键优化！）：
    • 为每个物理节点生成 K 个虚拟节点（如 node_i#0, node_i#1, …, node_i#K-1）；
    • 分别哈希后映射到环上；
    • 作用：显著改善哈希环上节点分布的均匀性，避免物理节点稀疏导致的负载倾斜。

无虚拟节点：3 个节点可能集中在环的 1/4 区域 → 负载偏差大
有虚拟节点（K=100）：300 个点均匀散布 → 负载标准差降低 5–10 倍

步骤 3：映射数据（Key）

  • 对任意数据 key，计算 h(key)，得到环上一点；
  • 顺时针查找：从 h(key) 出发，找到环上第一个（最近的）节点哈希值；
  • 该节点即为 key 的归属节点（存储/服务节点）。

为什么是“顺时针”？
约定方向，保证确定性；逆时针亦可，但需全局统一。

步骤 4：节点变更处理

场景 影响范围 迁移数据比例（理想）
新增节点 X X 的前驱节点（逆时针最近）到 X 之间的所有 key ≈ 1/(N×K)（K 为虚拟节点数）
删除节点 Y Y 所负责的全部 key ≈ 1/(N×K)
节点故障/网络分区 同删除节点 可结合副本机制（如主从）保障可用性

关键洞察：迁移量与总节点数 N 成反比，与新增/删除节点数无关 —— 实现可预测、可伸缩的变更成本。

三、核心优势（优点）

优势 说明 实际价值
极低的数据迁移开销 节点变更仅影响局部 key，避免全量 rebalance 降低运维风险、减少延迟抖动、提升系统稳定性（如缓存雪崩防护）
天然支持弹性伸缩 无需停服、无需预估容量，可动态扩缩容 云原生架构基石（如 Serverless、自动伸缩组）
负载相对均衡（尤其启用虚拟节点后） 虚拟节点将物理节点“打散”，提升哈希环覆盖密度 避免热点节点，提升资源利用率
实现简单，无中心依赖 仅需客户端或轻量代理维护环结构，无全局协调 降低系统复杂度与单点故障风险
兼容多种一致性模型 可与副本策略（如 NWR）、读写分离、多数据中心部署结合 支撑高可用、低延迟、异地多活等高级场景

四、固有局限与挑战（缺点）

缺点 原因分析 应对方案
无法绝对均衡（尤其小规模集群） 哈希函数的统计随机性 + 物理节点数量少 → 虚拟节点仍可能聚集 增大虚拟节点数（K=100~200）；使用更优哈希（如 Jump Hash、Rendezvous Hash）；运行时动态调优（如负载感知重分布）
环结构维护开销 节点变更需同步更新所有客户端的环信息，存在短暂不一致 引入服务发现（ZooKeeper/Etcd/Nacos）；客户端拉取+本地缓存+版本号校验；使用 gossip 协议传播变更
单点故障影响局部数据 若某节点宕机，其负责的 key 全部失效（无副本时） 必须配合副本机制（如主从、纠删码）；实现故障自动摘除与快速恢复
范围查询（Range Query）不友好 key 按哈希分布，逻辑相邻 key 可能落在不同节点 → 难以高效扫描 改用有序分片（如按 key 字典序分片）；引入二级索引；业务层聚合
哈希冲突与环分裂风险 极端情况下哈希碰撞导致节点重叠；环被“割裂”影响查找效率 使用高质量哈希函数；环结构采用平衡树/跳表优化查找（O(log N)）；客户端缓存 + 一致性校验
警告

注意：一致性哈希 ≠ 强一致性！它解决的是“数据路由一致性”，与 CAP 中的 Consistency（强一致）无关。

五、典型应用场景

场景 说明
分布式缓存（如 Redis Cluster） key 自动分片到不同 Redis 实例；扩容时仅迁移部分 slot（对应哈希段）
分布式数据库分片（如 Vitess、CockroachDB） 将表/行按 key 分布到不同存储节点，支持水平扩展
CDN 与边缘计算节点路由 用户请求哈希后定位最近边缘节点，节点增减不影响大部分用户
负载均衡器（如 Nginx+Consistent Hash） 将客户端 IP / Session ID 映射到后端服务，保障会话粘性
P2P 网络（如 Chord、Kademlia） 节点与数据均映射至环，实现去中心化查找
