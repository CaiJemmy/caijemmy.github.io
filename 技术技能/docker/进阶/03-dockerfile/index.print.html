03 Dockerfile

一、Dockerfile 是什么？

定义

Dockerfile 是一个文本文件，包含了一系列指令(instructions)，用于自动化构建 Docker 镜像。

类比理解

1Dockerfile : Docker镜像  ≈  Recipe : Cake
概念 类比 说明
Dockerfile 食谱 告诉厨师如何制作蛋糕
Docker 镜像 蛋糕模具 可以重复使用的模板
Docker 容器 具体的蛋糕 根据模具制作的实际蛋糕

工作流程

1编写 Dockerfile → docker build → Docker 镜像 → docker run → Docker 容器
2       ↓                ↓              ↓           ↓           ↓
3     配方           烘焙制作         蛋糕模具      制作蛋糕     具体蛋糕

二、Dockerfile 核心概念

1. 镜像的分层结构

Docker 镜像采用分层存储，每一条指令都会创建一个新的镜像层。

1# 示例 Dockerfile
2FROM ubuntu:20.04           # 基础层
3RUN apt-get update          # 第1层
4RUN apt-get install -y nginx # 第2层
5COPY app /var/www/html      # 第3层
6CMD ["nginx", "-g", "daemon off;"] # 第4层

分层的好处：

  • 缓存机制：未改变的层可以复用
  • 共享存储：多个镜像可以共享基础层
  • 快速构建：只需构建变化的层
  • 节省空间：避免重复存储相同内容

2. 构建上下文 (Build Context)

构建时，Docker 会将当前目录（或指定目录）的所有文件发送给 Docker 守护进程。

1# 当前目录的所有文件都会发送给 Docker
2docker build -t myapp:latest .
3
4# 只发送特定目录的文件
5docker build -t myapp:latest ./app
6
7# 使用远程 Git 仓库
8docker build -t myapp:latest https://github.com/user/repo.git

注意事项：

1# 1. 创建 .dockerignore 文件忽略不需要的文件
2# .dockerignore 内容：
3.git
4node_modules
5*.log
6*.tmp
7
8# 2. 构建上下文过大会影响构建速度，只构建需要的文件

三、Dockerfile 指令详解

Dockerfile官方参考

一、构建类指令（Build-time Instructions）

特点：仅在 docker build 过程中执行，用于构建镜像层（layer），影响镜像内容和结构；不会在容器启动时重复执行。 常见指令包括：

指令 说明 语法 示例
FROM 指定基础镜像（必须为第一条非注释指令） FROM <image>[:<tag>] [AS <name>] FROM nginx:1.21 AS builder
RUN 执行命令（如安装软件、编译代码），生成新镜像层 RUN <command>
RUN [“executable”, “param1”, “param2”]		 RUN apt-get update \
&& apt-get install -y \
python3 \
python3-pip \
nginx \
&& apt-get clean \
&& rm -rf /var/lib/apt/lists/*
COPY 将本地文件/目录复制到镜像中（推荐用于静态资源） COPY [--chown=<user>:<group>] <源路径>… <目标路径>
COPY [--chown=<user>:<group>] ["<源路径>",… “<目标路径>”]		 COPY --chown=node:node . /app
ADD 类似 COPY，但支持自动解压 tar 包、远程 URL（不推荐用于 URL，存在安全与缓存问题） ADD [--chown=<user>:<group>] <源路径>… <目标路径>
ADD [--chown=<user>:<group>] ["<源路径>",… “<目标路径>”]		 ADD app.tar.gz /app/
ADD https://example.com/app.tar.gz /tmp/
ARG 定义构建参数（需配合 --build-arg 使用，作用域为构建阶段） ARG <name>[=<default value>] docker build –build-arg VERSION=20.04 -t myapp:latest .
ONBUILD （已不推荐）为后续基于该镜像构建的子镜像预设触发指令

提示：RUN、COPY、ADD 是镜像分层的关键，每条指令通常生成一个只读层。

二、设置类指令（Configuration / Runtime Instructions）

特点：不参与镜像内容构建，而是声明容器运行时的配置或默认行为；在 docker run 时生效（可被 docker run 参数覆盖）。 常见指令包括：

指令 说明 语法 示例
ENV 设置环境变量（构建时和运行时均生效，可被 RUN 使用，也影响容器启动环境） ENV <key> <value>
ENV <key>=<value> …		 ENV \
NODE_ENV=production \
APP_PORT=3000 \
DB_HOST=localhost
EXPOSE 声明容器监听的端口（仅文档作用，不自动映射；实际映射需 docker run -p） EXPOSE <port> [<port>/<protocol>…] EXPOSE 80 443 3000
VOLUME 声明挂载卷（创建匿名卷或指定挂载点，影响容器运行时存储行为）创建挂载点，用于持久化数据 VOLUME ["/path/to/dir"]
VOLUME /path/to/dir		 VOLUME /data
VOLUME /usr/share/nginx/html
WORKDIR 设置工作目录（后续 RUN/CMD/ENTRYPOINT 等指令的默认路径）设置工作目录，相当于 cd WORKDIR /path/to/workdir # 复制到 /var/www/html
WORKDIR /var/www/html
COPY . .
USER 指定运行容器进程的用户（影响权限，提升安全性） USER <user>[:<group>]
USER <UID>[:<GID>]		 RUN groupadd -r app && useradd -r -g app appuser
USER appuser
COPY --chown=appuser:appuser . /app
LABEL 为镜像添加元数据（如版本、作者），不改变运行行为 LABEL <key>=<value> <key>=<value> … LABEL version=“1.0”
LABEL description=“My Web Application”
CMD 提供容器启动时的默认执行命令（可被 docker run 命令行覆盖） CMD [“executable”,“param1”,“param2”]）
CMD command param1 param2
CMD [“param1”,“param2”]		 # CMD 与 ENTRYPOINT 配合使用
ENTRYPOINT [“docker-entrypoint.sh”]
CMD [“nginx”, “-g”, “daemon off;”]
# 运行：docker run myimage
# 执行：docker-entrypoint.sh nginx -g “daemon off;”
ENTRYPOINT 配置容器的可执行入口（更常用于定义容器的主程序，与 CMD 协同使用） ENTRYPOINT [“executable”, “param1”, “param2”]
ENTRYPOINT command param1 param2
SHELL 指定 RUN/CMD/ENTRYPOINT 的默认 shell（如 ["/bin/bash", "-c"]）

注意：ENV 和 WORKDIR 在构建阶段也生效（如 RUN 中可使用），但其核心作用是为运行时提供配置，因此归入设置类更符合语义。

补充说明（非指令，但需注意）：

  • #：注释（非指令，不执行）
  • HEALTHCHECK：定义健康检查（属于运行时配置，可归入设置类）
  • STOPSIGNAL：指定容器停止时发送的系统信号（运行时配置）。当执行 docker stop <container> 时，Docker 守护进程默认向容器内 PID 1 进程 发送 SIGTERM 信号（等待默认 10 秒），若进程未退出，再发 SIGKILL 强制终止。而 STOPSIGNAL 指令的作用是：覆盖这个默认的 SIGTERM，改用你指定的信号。
    • STOPSIGNAL 只影响 docker stop 行为，不影响 docker kill（后者默认发 SIGKILL，除非显式指定 -s）；
    • 它不改变进程自身行为，是否响应该信号完全取决于应用代码（需主动 signal() 或 trap）；
    • 信号始终发给 PID 1 进程（即容器主进程），不是 shell 或 wrapper 脚本（除非你用 exec 正确传递）。
  • SHELL（已列）：影响指令执行方式，属配置类

HEALTHCHECK - 健康检查

作用：定义容器健康检查

 1# 语法
 2HEALTHCHECK [OPTIONS] CMD command
 3HEALTHCHECK NONE  # 禁用
 4
 5# 选项
 6--interval=DURATION      # 检查间隔（默认30s）
 7--timeout=DURATION       # 超时时间（默认30s）
 8--start-period=DURATION  # 启动等待时间（默认0s）
 9--retries=N              # 重试次数（默认3）
10
11# 示例
12HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
13  CMD curl -f http://localhost:8080/health || exit 1
14
15HEALTHCHECK CMD pg_isready -U postgres || exit 1
16
17HEALTHCHECK --interval=5m --timeout=3s \
18  CMD python healthcheck.py || exit 1
19
20# 查看健康状态
21docker ps
22# CONTAINER ID   STATUS
23# abc123        Up 2 minutes (healthy)

分类总结表：

COPY vs ADD —— 文件复制指令

维度 COPY ADD
核心功能 仅支持本地文件/目录复制到镜像内指定路径。 支持本地文件/目录复制，额外支持：
• 远程 URL（如 ADD https://...）
• 自动解压 .tar、.tar.gz、.tgz、.bz2、.xz 等归档文件（解压到目标路径）
语义清晰性 语义单一、明确：“复制即复制”，无隐式行为。 语义复杂：同一指令可能触发复制、下载、解压，易引发意外行为（如误解压非 tar 文件、URL 失败静默忽略等）。
安全性与可重现性 更安全：
• 不访问网络（无 URL 风险）
• 不执行解压（避免恶意归档或解压路径遍历）
• 构建可完全离线、确定性更强。		 风险点：
• ADD URL 依赖网络，破坏构建可重现性；
• 自动解压可能被利用（如 CVE-2019-14271 类漏洞）；
• 解压行为不可控（如权限、符号链接处理）。
Docker 官方建议 推荐首选（Docker 官方文档明确指出：“COPY is preferred”）。 仅在必需时使用：如需自动解压本地 tar 包（且确认来源可信、格式安全）。
典型用例 COPY ./src /app/src
COPY package.json /app/		 # 偶尔用于解压可信构建依赖（不推荐）
ADD ./deps.tgz /usr/local/lib/
# 避免：
# ADD https://example.com/app.tar.gz /app/ # 网络依赖、不可重现

最佳实践总结：

始终优先使用 COPY；仅当明确需要 ADD 的自动解压能力（且 tar 包来自可信构建上下文、非网络）时才使用 ADD。禁止在生产 Dockerfile 中使用 ADD 下载远程文件（应改用 RUN curl/wget && tar -x 显式控制）。

CMD vs ENTRYPOINT —— 容器启动指令

维度 CMD ENTRYPOINT
核心作用 提供容器默认执行命令及参数，可被 docker run 命令行完全覆盖。 配置容器的可执行入口点，docker run 传入的参数追加到 ENTRYPOINT 之后（exec 模式下），形成最终命令。
覆盖行为 CMD [“nginx”, “-g”, “daemon off;”]
# 运行时：
docker run myimg bash # 执行 bash（完全替换 CMD）		 ENTRYPOINT [“nginx”, “-g”]
CMD [“daemon off;”]
# 运行时：
docker run myimg # nginx -g “daemon off;"
docker run myimg -t # nginx -g “-t”（CMD 被替换，参数追加）
语法形式 支持两种形式：
• exec 形式（推荐）：CMD ["executable","param1"]（避免 shell 解析，PID 1 为进程本身）
• shell 形式：CMD command param1（实际执行 /bin/sh -c "command param1"，PID 1 为 /bin/sh）		 同样支持 exec/shell 形式，但 exec 形式是唯一推荐方式（确保容器信号（如 SIGTERM）能正确传递给主进程）。
协作关系 常与 ENTRYPOINT 配合使用：
• ENTRYPOINT 定义可执行程序（固定部分）
• CMD 提供默认参数（可被覆盖的灵活部分）		 是容器行为的“骨架”，定义容器本质用途（如 ENTRYPOINT ["redis-server"] 表示这是一个 Redis 容器）。
典型场景 • 提供默认运行命令（如 CMD ["node", "app.js"]）
• 作为 ENTRYPOINT 的默认参数补充		 • 构建可执行镜像（如 docker run mytool --help）
• 强制容器以特定程序启动（如 ENTRYPOINT ["/docker-entrypoint.sh"] 配合初始化脚本）
• 实现 CLI 工具镜像（如 docker run -v $(pwd):/data alpine tar -cf archive.tar /data）
陷阱警示 CMD 单独使用时，若用户 docker run 指定命令，整个 CMD 被丢弃（无继承）。 ENTRYPOINT 使用 shell 形式（ENTRYPOINT command）会导致 PID 1 为 /bin/sh，无法接收系统信号（如 docker stop 超时强制 kill）→ 必须用 exec 形式。

最佳实践总结：

  • ENTRYPOINT 定义容器“身份”（如 ["/bin/myapp"]），CMD 提供默认“行为”（如 ["--config", "/etc/myapp.conf"]）；
  • 始终使用 exec 形式（["binary", "arg"]），避免 shell 形式；
  • 若需初始化逻辑（如环境检查、配置生成），用 ENTRYPOINT 指向 shell 脚本（如 ["/docker-entrypoint.sh"]），并在脚本末尾用 exec "$@" 将控制权交还给 CMD；
  • 单一用途镜像优先设 ENTRYPOINT，通用基础镜像可仅设 CMD。
类别 执行时机 是否影响镜像层 典型用途
构建类 docker build 是 安装软件、复制文件、编译代码
设置类 docker run 否（仅配置） 设置环境、用户、端口、启动命令等

四、Dockerfile 最佳实践

1. 多阶段构建

 1# 阶段1：构建阶段
 2FROM node:16 AS builder
 3WORKDIR /app
 4COPY package*.json ./
 5RUN npm ci
 6COPY . .
 7RUN npm run build
 8
 9# 阶段2：生产阶段
10FROM nginx:alpine
11COPY --from=builder /app/dist /usr/share/nginx/html
12EXPOSE 80
13CMD ["nginx", "-g", "daemon off;"]
14
15# 优点：
16# 1. 最终镜像只包含运行时文件
17# 2. 减小镜像大小
18# 3. 提高安全性（不包含构建工具）

2. 最小化镜像大小

 1# 不好：镜像很大（~1GB）
 2FROM ubuntu:20.04
 3RUN apt-get update && apt-get install -y python3
 4
 5# 好：使用轻量基础镜像（~100MB）
 6FROM python:3.9-slim
 7
 8# 更好：使用 Alpine Linux（~15MB）
 9FROM python:3.9-alpine
10
11# 清理技巧
12RUN apt-get update \
13    && apt-get install -y \
14        build-essential \
15        python3-dev \
16    && pip install --no-cache-dir -r requirements.txt \
17    && apt-get remove -y build-essential python3-dev \
18    && apt-get autoremove -y \
19    && apt-get clean \
20    && rm -rf /var/lib/apt/lists/*

3. 优化构建缓存

 1# 不好：缓存无效
 2COPY . /app
 3RUN npm install
 4
 5# 好：利用缓存
 6# 1. 先复制 package.json（不经常变化）
 7COPY package*.json ./
 8RUN npm install
 9
10# 2. 再复制源代码（经常变化）
11COPY . .
12
13# 原理：如果 package.json 没变化，npm install 会使用缓存

4. 安全性最佳实践

 1# 1. 使用非 root 用户
 2RUN addgroup -g 1000 -S appuser && \
 3    adduser -u 1000 -S appuser -G appuser
 4USER appuser
 5
 6# 2. 定期更新基础镜像
 7FROM ubuntu:20.04
 8RUN apt-get update && apt-get upgrade -y
 9
10# 3. 扫描漏洞
11# 构建后运行：docker scan myimage:latest
12
13# 4. 不包含敏感信息
14# 不好：在 Dockerfile 中写密码
15ENV DB_PASSWORD=secret123
16
17# 好：通过环境变量或 secrets 传递
18docker run -e DB_PASSWORD=secret123 myimage
